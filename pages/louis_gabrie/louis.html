What I Worked On

Summary

Things to know before reading:
- Jira is a productivity tool that breaks takes down tasks, features, and bugs into issues and an epic is a collection of this issues.
- A feature branching workflow is used at ALICE, with environments to test the features made
- Features are made in feature branches, which are binded to issues. These issues are then binded into certain epics, whereas they each epic has their own branch.

The major problem presented is that at ALICE, we are uncertain that the features/bug-fixes binded to a Epic made it to the Epic environment for testing, and is qualified to be there. A high certainty that these features/bug-fixes binded to a Epic branch made it to the Epic environment for testing is critical in keeping clients happy, while minimizing the amount of time wasted double checking the Epic for code complete, or while testing an unstable Epic environment. Issues should be linked with code and vice versus, giving us the certainty that all code is tracked in Jira. Along with that, all issues statuses should match the statuses of their code, giving us the certainty that no code is left behind.

In order to handle this problem, a web application called branch-monitor was built, to decrease the amount of time allotted to managing the stability of a epic environment, while giving the user reassurance that all Jiras have code and their designated items in the stage they are at in their life cycle. Worked with a partner and mainly worked with a software stack called, MEAN. The software stack allowed us to handle real time updates to the user interface from the Bitbucket and Jira Cloud RESTful APIs to deliver information about a particular epic to the user.
